## 1. 개요

### 1.1 한 줄 소개

웹 브라우저 기반 실시간 전략(RTS) — HTML, CSS, JS 리소스를 채집해 유닛을 생산하고 상대 웹 페이지를 점령하는 멀티플레이어 게임

### 1.2 게임 컨셉

플레이어는 자신의 "웹 페이지"를 기지로 삼는다. 웹을 구성하는 3대 요소(HTML, CSS, JavaScript)가 곧 전략 자원이다. 이 자원으로 유닛을 생산하고, 상대 페이지를 공격해 자원을 탈취하거나 점령한다. 웹 개발의 메타포를 전략 게임에 녹여 **"코드로 전쟁한다"** 는 독특한 판타지를 제공한다.

### 1.3 타겟 플레이어

| 구분 | 프로필 |
|------|--------|
| **1차 타겟** | 웹 개발에 관심 있는 10~30대, .io 게임 유저 |
| **2차 타겟** | 캐주얼 RTS/전략 게임 유저, 브라우저 게임 유저 |
| **세션 패턴** | 점심시간, 쉬는 시간 등 10~15분 마이크로 세션 |

### 1.4 벤치마크 분석

| 게임 | 장르 | 참고 요소 |
|------|------|-----------|
| **Slither.io / Agar.io** | .io 멀티플레이어 | 즉시 접속, 짧은 세션, 바이럴 구조 |
| **Lordz.io** | .io RTS | 브라우저 RTS의 간소화 모델 |
| **Territorial.io** | 영토 전략 | 영토 점령 메카닉 |
| **StarCraft** | 전통 RTS | 가위바위보 상성, 경제 설계 |
| **Age of Empires** | 전통 RTS | 리소스 시스템, 유닛 카운터 |

### 1.5 차별화 포인트

1. **웹 메타포**: HTML/CSS/JS라는 현실 소재를 게임 자원으로 활용 → 개발자 커뮤니티 자연 유입
2. **제로 설치**: 브라우저만으로 즉시 플레이 → .io 게임 수준의 접근성
3. **웹 페이지 점령**: 가상의 "웹 페이지"를 점령/방어하는 독특한 영토 시스템
4. **코드 = 전략**: 웹 기술 이름을 활용한 스킬/유닛으로 학습과 재미 동시 제공

## 2. 게임 디자인 프레임워크

### 2.1 MDA 분석

> Hunicke, LeBlanc & Zubek (2004) — Mechanics → Dynamics → Aesthetics

| 레이어 | 설계 |
|--------|------|
| **Mechanics** | 리소스 채집, 유닛 생산, 가위바위보 상성, 웹 페이지 점령 |
| **Dynamics** | 리소스 배분 딜레마, 상성 읽기, 공방 타이밍 판단, 연합 형성 |
| **Aesthetics** | **Challenge**(전략적 도전) + **Fellowship**(협동/경쟁) + **Discovery**(상성 조합 발견) + **Fantasy**(웹 해커 판타지) + **Expression**(전략 조합/기지 커스터마이징) + **Sensation**(Juice 이펙트) |

목표 미학: Challenge와 Fellowship을 중심축으로, Fantasy(웹 해킹)로 테마 몰입 극대화. Expression은 전략 조합 자유도 + 기지/유닛 커스터마이징으로, Sensation은 전투 Juice 이펙트로 충족. 타겟의 "쉬는 시간 플레이" 패턴에 맞춰 Submission(여가) 미학도 부분적으로 지원.

### 2.2 플로우 이론 적용

> Csikszentmihalyi (1990) — 도전 ≈ 능력 → 몰입

```
불안 영역 (너무 어려움)
  ↑
  |         ★ FLOW ZONE ★
도전 |       /
수준 |     /  ← DDA로 이 채널 유지
  |   /
  | /
  +―――――――――――→ 플레이어 능력
       지루함 영역 (너무 쉬움)
```

| 플로우 요소 | 게임 내 적용 |
|-------------|-------------|
| **명확한 목표** | "상대 페이지를 점령하라" — 매 라운드 명시적 목표 |
| **즉각적 피드백** | 자원 획득 숫자 팝업, 유닛 충돌 이펙트, 점령 게이지 |
| **도전-능력 균형** | ELO 기반 매칭 + DDA(후술) + 난이도 선택(봇 Easy/Normal/Hard) |
| **행동-인식 통합** | 클릭 → 유닛 반응 100ms 이내, 드래그 선택의 직관적 조작감 |
| **주의 집중** | 전투 중 불필요 알림 차단, UI 자동 숨김, 채팅 최소화 |
| **통제감** | 유닛 배치/이동 직접 지시, 리소스 배분 자율 선택 |
| **자의식 상실** | 몰입 파괴 요소 제거: 팝업 광고 없음, 강제 이벤트 없음 |
| **시간 왜곡** | 10~15분 세션이지만 체감 5분 → 짧은 라운드 연속 |
| **자기목적적 경험** | 외적 보상 없이도 전략 실험 자체가 재미 — 연습 모드 제공 |

### 2.3 자기결정이론 (SDT) 적용

> Deci & Ryan (1985) — 자율성, 유능감, 관계성

| SDT 욕구 | 충족 설계 |
|----------|----------|
| **자율성** | 유닛 조합 자유 선택, 공격/방어/확장 전략 자율 결정, 리소스 배분 |
| **유능감** | 랭킹 상승, 승리 피드백, "코드 마스터" 칭호 시스템, 스킬 언락 |
| **관계성** | 실시간 대전, 연합(길드), 전투 후 리플레이 공유, 채팅 |

### 2.4 플레이어 유형별 설계

> Bartle (1996) — Achiever, Explorer, Socializer, Killer

| 유형 | 충족 요소 |
|------|----------|
| **Achiever** | 시즌 랭킹, 업적 시스템, 유닛 해금 진행도 |
| **Explorer** | 새로운 상성 조합 발견, 숨겨진 유닛 조합(시너지), 맵 탐색 |
| **Socializer** | 연합 시스템, 채팅, 전투 결과 공유, 협동 공격 |
| **Killer** | PvP 점령전, 랭킹 경쟁, 자원 약탈 |

> **유형 교차 설계 (Quantic Foundry 참조):** 플레이어는 단일 유형이 아닌 비율 조합. Explorer+Achiever를 위한 "숨겨진 업적"(비밀 유닛 조합 발견 시 해금), Socializer+Killer를 위한 "연합 PvP 토너먼트", Achiever+Explorer를 위한 "마스터리 챌린지"(후술) 등 교차 콘텐츠 설계.

### 2.5 세계관 (Narrative)

> MDA Narrative Aesthetic — 최소한의 세계관으로 테마 몰입 강화

**"인터넷이 전쟁터가 된 세상."**

웹의 구조가 무너지기 시작했다. HTML의 뼈대가 흔들리고, CSS의 껍질이 벗겨지며, JavaScript의 로직이 폭주한다. 플레이어는 **웹 마스터**로서 자신의 도메인을 지키고, 타락한 페이지를 정화하며, 영토를 확장해야 한다.

- **시즌 내러티브**: 시즌별 간단한 스토리 이벤트 (예: "DDoS 대란 시즌", "레거시 코드의 습격")
- **미션 브리핑**: 각 라운드 시작 시 한 줄 브리핑으로 상황 몰입 ("적 서버가 당신의 API를 노리고 있습니다")

## 3. 핵심 게임 루프

### 3.1 코어 루프 (1라운드: 10~15분)

```
┌─────────────────────────────────────────────────┐
│                                                 │
│   ① 채집 ──→ ② 생산 ──→ ③ 전투 ──→ ④ 점령     │
│      ↑                                  │       │
│      └──────────── 보상 ←───────────────┘       │
│                                                 │
└─────────────────────────────────────────────────┘
```

| 단계 | 행동 | 시간 비중 |
|------|------|----------|
| **① 채집** | 웹 페이지에서 HTML/CSS/JS 리소스 자동 수집 + 수동 부스트 | 상시 (백그라운드) |
| **② 생산** | 리소스 소비 → 유닛 생산, 업그레이드 선택 | 전략 판단 (~30%) |
| **③ 전투** | 유닛 편성 → 상대 페이지 공격 or 방어 배치 | 액션 (~50%) |
| **④ 점령** | 전투 승리 → 상대 페이지 점령 → 추가 자원 확보 | 보상 (~20%) |

> **세션 설계 근거:** 브라우저 게임 최적 세션 10~15분 (GDevelop, GameDistribution 리서치). 점심시간/쉬는 시간에 1~2판 가능.

### 3.2 메타 루프 (세션 간)

```
라운드 승리 → 경험치/랭크 → 새 유닛·스킬 해금 → 더 강한 전략 → 다음 라운드
```

- **시즌제** (4주 단위): 시즌 랭킹 리셋, 시즌 보상
- **수집**: 유닛 스킨, 칭호, 이모트 수집
- **연합**: 연합 랭킹, 연합전

### 3.3 Action → Reward → Investment 사이클

> 캐주얼 게임 루프 이론 (GDevelop, 2025)

| 사이클 | 내용 |
|--------|------|
| **Action** | 리소스 채집, 유닛 전투 |
| **Reward** | 점령 보상, 경험치, 랭크 포인트, 드롭 아이템 |
| **Investment** | 유닛 업그레이드, 새 전략 해금, 기지 강화 |

## 4. 리소스 시스템

### 4.1 3대 리소스

| 리소스 | 아이콘 | 역할 | 채집원 |
|--------|--------|------|--------|
| **HTML** `</>` | 주황 | 기본 유닛 생산, 구조물 건설 | 기지 자동 생성 + 점령 페이지 |
| **CSS** `{ }` | 파랑 | 방어/버프 유닛 생산, 기지 강화 | 기지 자동 생성 + 점령 페이지 |
| **JavaScript** `( )` | 노랑 | 공격/특수 유닛 생산, 스킬 사용 | 기지 자동 생성 + 점령 페이지 |

### 4.2 리소스 경제 설계

> StarCraft II, Age of Empires II 경제 모델 참조

#### 채집 구조

```
기지 (Base Page)
├── HTML 생성기: 2/sec (기본)
├── CSS 생성기:  1.5/sec (기본)
└── JS 생성기:   1/sec (기본)

점령 페이지 (Captured Page) — 각 +50% 보너스
```

#### 핵심 경제 공식

```
총 수입 = 기지_기본_수입 × (1 + 점령_페이지_수 × 0.5) × 업그레이드_배율

유닛 비용 = 기본_비용 × (1 + 인플레이션_계수 × 라운드_경과_시간)
```

#### 리소스 밸런스 삼각형

```
        HTML (구조/체력)
       /              \
      /    균형 지점    \
     /                  \
CSS (방어/버프) ──── JS (공격/특수)
```

- **HTML 올인**: 물량 전략 — 저렴한 유닛 대량 생산
- **CSS 올인**: 거북이 전략 — 강력한 방어, 상대가 지치길 기다림
- **JS 올인**: 러시 전략 — 고화력 소수 유닛으로 기습

> **설계 원칙:** Supreme Commander의 Flow 경제 모델 참조. 리소스는 저장 상한이 있으며(각 500), 사용하지 않으면 넘치므로 **지속적 소비**를 유도. 이는 적극적 플레이를 장려하고 터틀링(수비만 하기)을 억제한다.

### 4.3 인플레이션 방지 설계

> Amazon New World 디플레이션 사태 교훈 적용

| 전략 | 구현 |
|------|------|
| **리소스 상한** | 각 리소스 최대 500. 넘치면 소실 |
| **유지 비용** | 유닛 보유 시 초당 소량의 리소스 소비 (인구수 비용) |
| **전투 소모** | 전투 시 유닛 체력 자연 감소 → 리소스로 수리 필요 |
| **라운드 제한** | 15분 라운드 → 자연스러운 경제 리셋 |

## 5. 유닛 시스템

### 5.1 기본 유닛 3종

> RTS 가위바위보 상성 모델 (StarCraft, AoE) 적용

| 유닛 | 리소스 | 역할 | HP | ATK | SPD | 특성 |
|------|--------|------|-----|-----|-----|------|
| **HTML 솔저** `<div>` | HTML 30 | 탱커/물량 | 120 | 8 | 1.0 | 저렴, 대량 생산 |
| **CSS 가디언** `{shield}` | CSS 40 | 방어/지원 | 80 | 5 | 0.8 | 아군 버프, 디버프 |
| **JS 스트라이커** `func()` | JS 50 | 딜러/기습 | 60 | 15 | 1.5 | 고화력, 저체력 |

### 5.2 가위바위보 상성 매트릭스

> StarCraft BW 사이즈 매트릭스 + AoE2 보너스 데미지 모델 절충

```
HTML 솔저 ──강함──→ JS 스트라이커
    ↑                     │
    │                     │
  강함                  강함
    │                     │
    │                     ↓
CSS 가디언 ←──강함── JS 스트라이커
    │
    ↓
HTML 솔저 ←──강함── CSS 가디언  ← (CSS가 HTML 디버프)
```

**정리:**
- **HTML 솔저 > JS 스트라이커**: 물량으로 압도 (데미지 흡수)
- **JS 스트라이커 > CSS 가디언**: 높은 화력이 방어 관통
- **CSS 가디언 > HTML 솔저**: 디버프로 물량 무력화

#### 데미지 계산 공식

```
실제_데미지 = (공격력 + 보너스_데미지) × 상성_배율 - 방어력

상성 배율:
  유리한 상성: 1.5x
  동일 상성:   1.0x
  불리한 상성: 0.7x
```

### 5.3 상위 유닛 (업그레이드 트리)

라운드 내 리소스를 모아 상위 유닛으로 업그레이드 가능:

```
Tier 1 (기본)          Tier 2 (중급)              Tier 3 (고급)
─────────────────────────────────────────────────────────────
HTML 솔저 <div>   →   HTML 나이트 <section>   →   HTML 타이탄 <main>
  HP 120, ATK 8        HP 200, ATK 12              HP 350, ATK 18
  비용: H30             비용: H60 + C20             비용: H100 + C40 + J20

CSS 가디언 {shield} → CSS 메이지 {animate}    →   CSS 아크메이지 {transform}
  HP 80, ATK 5         HP 130, ATK 8, 범위버프     HP 180, ATK 12, 광역버프
  비용: C40             비용: C80 + H20             비용: C120 + H40 + J20

JS 스트라이커 func() → JS 어쌔신 async()      →   JS 오버로드 Promise.all()
  HP 60, ATK 15        HP 90, ATK 22, 은신         HP 120, ATK 35, 광역
  비용: J50             비용: J90 + H20             비용: J140 + H40 + C20
```

> **설계 원칙:** Tier 2부터 복합 리소스 필요 → 경제 다각화 강제. 순수 원리소스 올인 전략에 상한 부여.

### 5.4 특수 유닛

| 유닛 | 비용 | 특수 능력 |
|------|------|----------|
| **API 커넥터** | H40+C40+J40 | 아군 유닛 간 시너지 버프 (인접 유닛 ATK +20%) |
| **Debug 드론** | J80 | 적 유닛 1기 즉사 (쿨타임 60초) |
| **Firewall** | C100+H50 | 고정 방어 구조물, 범위 내 적 이동속도 -50% |

### 5.5 유닛 밸런스 원칙

> "Counter Intensity" — Dawn of War 2 스타일 소프트 카운터 채택

1. **상성은 결정적이지만 절대적이지 않다**: 1.5x 배율이므로 수적 우위로 극복 가능
2. **Tier 차이는 상성을 뒤집을 수 있다**: Tier 2 불리 상성 > Tier 1 유리 상성
3. **혼합 편성이 최적**: 단일 유닛 올인은 카운터에 취약
4. **포지셔닝 보너스**: 측면/후방 공격 시 +20% 추가 데미지

### 6.4 쿨다운 리듬 설계

> PLAY 휴리스틱 (Desurvire, 2013) — 전투 사이 인지적 휴식 필요

```
전투 종료 → [전황 정리 10초] → 리소스 재분배 → 다음 전투 준비
              ├── 획득 자원 요약
              ├── 소실 유닛 표시
              └── 다음 행동 추천
```

전투 후 10초간 "전황 정리" 화면을 표시하여 플레이어에게 인지적 휴식을 제공. 이 시간 동안 리소스 재분배와 유닛 재편성이 가능.

### 6.5 동적 난이도 조절 (DDA)

> Mortazavi et al. (2024), Zohaib (2018) — DDA는 enjoyment, flow에 유의미한 효과

| 측정 지표 | DDA 반응 |
|----------|---------|
| 연패 3회 이상 | 매칭 풀 하향, 상대 약간 약한 유저 배정 |
| 연승 3회 이상 | 매칭 풀 상향, 보상 약간 감소 |
| 라운드 내 전투 무승부 | 중립 페이지 리소스 +10% (전황 촉진) |
| 한쪽 압도적 우세 | 열세 측 유닛 수리 비용 -20% (은밀한 보조) |

> **핵심 원칙:** DDA는 플레이어가 인지하면 부정적 반응 유발. 모든 조절은 매칭 단계에서 수행하거나 자연스러운 게임 메카닉(중립 자원 변동)으로 위장.

### 7.1 보상 스케줄

> Skinner 강화 스케줄 이론 적용

### 7.2 에스컬레이팅 보상 곡선

> ihobo (2011) — 초반 보상 풍성 → 점점 희귀

| 레벨 구간 | 필요 XP | 레벨업 시 보상 | 체감 |
|----------|---------|-------------|------|
| **1~5** | 각 100 XP | 매 레벨 유닛 스킨 + 코인 200 | 빠른 성장감 (훅) |
| **6~15** | 각 250 XP | 격 레벨 스킨, 매 레벨 코인 150 | 안정적 보상 |
| **16~30** | 각 500 XP | 5레벨마다 특별 보상, 매 레벨 코인 100 | VR 드롭 + 시즌 보상으로 동기 유지 |
| **31~50** | 각 1000 XP | 10레벨마다 한정 칭호 | 마스터리 자체가 보상 |

#### "한 판 더" 심리 설계

- **패배 시**: "거의 이길 뻔했습니다! 상대 기지 HP 12% 남음" 피드백
- **연패 시**: "다음 판 첫 승 보너스 3배!" 복귀 인센티브
- **승리 직전 끊김**: 재접속 시 마지막 라운드 결과 + "아까 못 받은 보상" 표시

#### 윤리적 보상 가이드라인

> 벨기에, 네덜란드 가챠 규제 동향 반영

- **유료 구매 + 변동비율(VR) 조합 금지**: 유료 아이템은 확정 구매만 가능
- **드롭 아이템은 게임 플레이로만 획득**: 돈으로 드롭 확률 구매 불가
- **확률 공개**: 모든 VR 드롭의 확률 테이블을 게임 내 공개

### 7.3 부여된 진행 효과 (Endowed Progress)

> Nunes & Dreze (2006) — 시작부터 진행된 느낌 제공

- 신규 가입 시 즉시 **HTML 솔저 5기** 지급

### 7.4 랭킹 시스템

```
기대 승률:  E_A = 1 / (1 + 10^((R_B - R_A) / 400))

```

|------|----------|------|
| 신규 (< 30판) | 40 | 빠른 실력 수렴 |
| 일반 (< 2000) | 24 | 안정적 변동 |
| 고수 (2000+) | 16 | 최소한의 변동 |

#### 랭크 티어

| **Platinum** | 1800 ~ 2199 | `<script>` |
| **Diamond** | 2200+ | `<fullstack/>` |

### 8.1 리텐션 목표

### 8.2 첫 사용자 경험 (FTUE)

> 인지 부하 이론 (Sweller, 1988) — 새 메카닉은 하나씩 도입

```
[접속] → [닉네임 입력] → [즉시 튜토리얼 전투] → [첫 승리!] → [보상 획득] → [첫 실전]
  0초      10초            30초~2분             2분         2분30초       3분
```

1. **0~10초**: 닉네임 입력만 요구 (회원가입 없음)
2. **10초~2분**: AI 상대 튜토리얼 — HTML 솔저만으로 승리
3. **2~3분**: 보상 획득 + CSS/JS 유닛 소개
4. **3분~**: 실제 매칭 (첫 3판은 비슷한 레벨 보장)

### 8.3 재방문 설계

> 재방문 5대 요소 (절차적 생성, 마스터리 곡선, 소셜 경쟁, 수집, 콘텐츠 업데이트) 전체 충족

| 요소 | 구현 | 재방문 이론 |
|------|------|-----------|
| **일일 보상** | 접속만 해도 랜덤 리소스 팩 지급 | 고정간격(FI) |
| **7일 스트릭** | 7일차에 프리미엄 스킨 | 손실 회피 |
| **시즌 패스** | 무료/프리미엄 트랙 (포트나이트 모델) | 콘텐츠 업데이트 |
| **주간 토너먼트** | 매주 특별 규칙 토너먼트 | 절차적 생성 |
| **연합전** | 연합 간 주간 랭킹 | 소셜 경쟁 |
| **마스터리 챌린지** | 승리 이상의 도전 목표 (7.5절) | 마스터리 곡선 |
| **유닛/칭호 수집** | 도감 완성 욕구 자극 | 수집 요소 |

### 8.4 IKEA 효과 — 기지 커스터마이징

> Norton et al. (2012) — 직접 만든 것에 더 높은 가치 부여

플레이어가 기지 페이지를 **직접 꾸미는** 시스템. 단순 프리셋 구매가 아니라 요소 조합:

| 커스터마이징 요소 | 내용 |
|----------------|------|
| **레이아웃** | 기지 내 구조물 배치 위치 직접 결정 (전략성 + 소유감) |
| **색상 테마** | 배경색, 강조색 직접 선택 |
| **방어 구조물 배치** | Firewall 위치를 직접 설정 → 전략적 IKEA 효과 |
| **기지 이름** | 커스텀 도메인 네임 (예: "my-fortress.dev") |

이를 통해 **"내가 만든 기지"** 에 대한 애착 형성 → 방어 동기 강화 → 리텐션 증가

### 9.4 인지 부하 관리

> Sweller (1988) — 작업 기억 7±2 항목 한계 / Schwartz (2004) — 선택의 역설

#### 점진적 공개 (Progressive Disclosure)

| 플레이어 경험 | HUD 표시 범위 |
|-------------|-------------|
| **첫 5판** | HTML 리소스만 + 기본 유닛 3종만 + 미니맵 숨김 |
| **6~10판** | 전체 리소스 3종 + Tier 2 해금 알림 |
| **11판~** | 풀 HUD + 미니맵 + 특수 유닛 해금 |
| **숙련자** | 설정에서 즉시 풀 HUD 활성화 가능 |

#### 선택의 역설 대응

> 한 번에 너무 많은 선택지 → 만족도 하락

- **생산 메뉴**: 현재 해금된 유닛만 표시, "추천" 태그로 최적 유닛 제안
- **전략 가이드**: 상대 편성 감지 → "CSS 가디언이 효과적입니다" 추천
- **초보 모드**: Tier 1 유닛 3종으로 제한, 복잡한 선택지 차단

### 9.5 게임 필 & Juice

> Swink (2008) — 3가지 기둥: 실시간 제어 + 시뮬레이션 공간 + 폴리시

#### 기둥 1: 실시간 제어 (Real-time Control)

| 항목 | 목표치 |
|------|--------|
| **입력 → 반응 지연** | 100ms 이내 |
| **유닛 선택 피드백** | 즉시 선택 원 표시 + 클릭 사운드 |
| **이동 명령 피드백** | 목적지 마커 + 경로 프리뷰 |
| **클라이언트 보간** | 서버 틱(50ms) 사이 부드러운 위치 보간 |

#### 기둥 2: 시뮬레이션 공간 (Simulated Space)

| 항목 | 구현 |
|------|------|
| **유닛 충돌** | 유닛 간 겹침 방지, 자연스러운 회피 이동 |
| **패스파인딩** | A* 기반, 장애물 자동 우회 |
| **이동 물리** | 가속/감속 곡선 적용 (급정지 없음) |

#### 기둥 3: 폴리시 (Juice) — 전체 체크리스트

| Juice 요소 | 적용 상황 | MVP 포함 |
|-----------|----------|---------|
| **화면 흔들림** | 전투 충돌, 점령 완료 | **Phase 1** |
| **파티클 이펙트** | 전투, 유닛 생산, 점령 | **Phase 1** |
| **데미지 숫자 팝업** | 공격 적중 시 | **Phase 1** |
| **색상 플래시** | 상성 유리(초록), 레벨업(골드), 피격(빨강) | **Phase 1** |
| **Squash & Stretch** | 유닛 이동 시 방향 스트레칭, 피격 시 스쿼시 | Phase 3 |
| **히트스톱/프리즈** | 상성 유리 적중 30ms, 결정적 킬 50ms | Phase 3 |
| **트레일 이펙트** | JS 스트라이커 이동 잔상, 스킬 코드 라인 | Phase 3 |
| **사운드 피치 랜덤** | 동일 공격 사운드 ±15% 피치 변화 | Phase 3 |
| **진동/햅틱** | 모바일: 전투, 점령, 레벨업 시 진동 패턴 | Phase 3 |

> **"Juice it or Lose it" 원칙 (GDC 2012):** 화면 흔들림, 파티클, 숫자 팝업, 색상 플래시의 핵심 4개는 **Phase 1부터 포함**. Juice 없는 프로토타입은 재미를 제대로 평가할 수 없다.

| 상황별 Juice 조합 |
|-----------------|
| **유닛 생산**: 팝업 애니메이션 + 코드 타이핑 사운드 + 숫자 카운트업 |
| **전투 충돌**: 화면 미세 흔들림 + 파티클 + 데미지 숫자 팝업 |
| **상성 유리 적중**: 색상 플래시(초록) + "SUPER EFFECTIVE!" + 히트스톱 |
| **점령 성공**: 게이지 채움 애니메이션 + 승리 사운드 + 화면 줌 |
| **레벨업**: 전체 화면 골드 플래시 + 팡파르 사운드 |
| **패배**: 화면 회색화 + 글리치 이펙트 (웹 테마) |

## 11. 기술 아키텍처

### 11.1 기술 스택

| 계층 | 기술 | 선택 이유 |
|------|------|----------|
| **프론트엔드** | Next.js + React + TypeScript | Vercel 최적화, SSR/SSG 지원 |
| **렌더링** | HTML5 Canvas (PixiJS) | 2D RTS에 최적, 경량 |
| **실시간 통신** | Colyseus | 델타 상태 동기화, 내장 매칭, 틱 기반 루프 |
| **데이터베이스** | PostgreSQL (Supabase) | 유저 데이터, 랭킹, 시즌 기록 |
| **인증** | Supabase Auth | 소셜 로그인 + 익명 로그인 |
| **배포 (프론트)** | Vercel | 글로벌 CDN, Edge Functions |
| **배포 (게임 서버)** | Railway / Fly.io | Colyseus 서버 호스팅 |
| **캐시** | Vercel KV (Redis) | 세션 캐시, 실시간 랭킹 |

### 11.2 아키텍처 다이어그램

```
┌─────────────────────────────────────────────────────────┐
│                      클라이언트                          │
│  Next.js (Vercel) + PixiJS (Canvas 렌더링)              │
│  └── Colyseus Client SDK (WebSocket 연결)               │
└────────────────────────┬────────────────────────────────┘
                         │ WebSocket
                         ▼
┌─────────────────────────────────────────────────────────┐
│                  Colyseus 게임 서버                       │
│  (Railway / Fly.io)                                     │
│  ├── Room: 매칭 + 게임 로직                              │
│  ├── State: 델타 동기화 (유닛, 리소스, 맵)                │
│  └── Tick Loop: 20 TPS (50ms 간격)                      │
└────────────────────────┬────────────────────────────────┘
                         │
              ┌──────────┼──────────┐
              ▼                     ▼
┌──────────────────┐   ┌──────────────────┐
│   Supabase        │   │   Vercel KV       │
│   (PostgreSQL)    │   │   (Redis)         │
│   - 유저 데이터    │   │   - 세션 캐시      │
│   - 랭킹 기록     │   │   - 실시간 리더보드 │
│   - 시즌 데이터    │   │   - 매칭 큐        │
└──────────────────┘   └──────────────────┘
```

### 11.3 네트워크 설계

| 항목 | 설계 |
|------|------|
| **프로토콜** | WebSocket (TCP) via Colyseus |
| **틱레이트** | 20 TPS (50ms) — 브라우저 RTS에 충분 |
| **상태 동기화** | 서버 권위(Server Authoritative) + 클라이언트 예측 |
| **지연 보상** | 입력 버퍼링 + 서버 리콘실리에이션 |
| **대역폭 최적화** | 델타 패치만 전송 (Colyseus 내장) |

### 11.4 매칭 시스템

```
매칭 요청 → Elo ±200 범위 검색 → 30초 내 미매칭 시 범위 확대(±400) → 매칭 성공 → Room 생성
```

- **매칭 시간 목표**: 30초 이내
- **Elo 범위**: 초기 ±200, 10초마다 ±100씩 확대
- **신규 유저 보호**: 첫 5판은 별도 신규 풀
